## 3. PROBLEMI STRUTTURALI (i veri killer)

### 🔴 P0 — Claude Engine V8 NON è collegato alla UI

Questo è il finding più grave dell'audit. Il "Claude Engine V8" descritto in `DIARIO_DI_BORDO_V8.md` come "tutto migrato" è in realtà **dead code**: i nuovi hook esistono ma nessun componente li importa.

```
useDownloadEngine     → 0 importatori (componenti)
useWcaJobs            → 0 importatori
useWcaAppDownload     → 1 (solo type import in DownloadProgressBar)
useWcaEnrich          → 0 importatori
useWcaVerify          → 0 importatori
useWcaPartners        → 0 importatori
useDirectoryDownload  → 0 importatori
```

**Cosa usa veramente la UI?**
```
ActionPanel.tsx
  └── useActionPanelLogic.ts
       └── scrapeWcaDirectory()  ← src/lib/api/wcaScraper.ts
            └── wcaDiscover()    ← src/lib/api/wcaAppApi.ts  ✅
```

Quindi il flusso reale di download passa per `wcaScraper.ts` (che internamente è migrato a `wcaAppApi.ts`), **non** per `useDownloadEngine` né per `useWcaJobs`. Tutto l'apparato V8 di hook è infrastruttura mai cablata.

**Impatto**: ~2.000 righe di codice non utilizzato, confonde Lovable, fa pensare che ci sia un sistema di job server-side quando in realtà si scarica ancora client-side.

**Risultato**: codice -2.500 LOC, mappa mentale chiara, voto stimato +500 punti.

### ▶ ONDATA 2 — RIDUZIONE FILE GIGANTI (settimana 2, ~10h)
**Obiettivo**: nessun file applicativo > 400 LOC.

### ▶ ONDATA 3 — TYPE SAFETY E LINT (settimana 3, ~8h)
**Obiettivo**: codice "intellisense-friendly" e bug latenti azzerati.

1. Abilitare `strict: true` in `tsconfig.app.json`. Fixare gli errori uno per file (saranno centinaia, ma molti si sistemano in batch).

**Risultato**: voto type-safety da 200 → 800. Voto lint da 150 → 750.

### ▶ ONDATA 4 — BACKEND CONSOLIDATION (settimana 4, ~12h)
**Obiettivo**: edge functions e tabelle ridotte di un terzo.

1. **Edge functions WCA**: deprecare le 9 vecchie (`scrape-wca-*`, `sync-wca-*`, `wca-auto-login`, `get-wca-credentials`, `save-wca-cookie`). Mantenere solo `process-download-job` e `save-wca-contacts` se ancora usate. Tutto il resto va a `wca-app.vercel.app`.

3. **Tabelle contatti**: progettare migrazione `imported_contacts` + `prospect_contacts` + `partner_contacts` → tabella `contacts` polimorfa con `entity_type`. Da fare in step DB-first con view di compatibilità.
4. **Coda download**: unificare `download_queue` e `download_jobs`/`download_job_items`/`download_job_events` in un singolo modello con `parent_id`.

**Risultato**: meno superfici di attacco, meno migration drift. Voto +800.

### ▶ ONDATA 5 — PERFORMANCE E TEST (settimana 5, ~8h)

2. **Test critici**: aggiungere test per

3. **Smoke test E2E** (Playwright o Cypress) per le 5 route principali.
4. **CI**: aggiungere GitHub Action che gira `tsc + eslint + vitest + build` su ogni PR.

**Risultato**: rotture in produzione → quasi zero. Voto +700.
